1L'architecture : le modèle MVC

Symfony range le code dans des dossiers selon son rôle. C'est le principe MVC : Modèle, Vue, Contrôleur.

🍽️ Analogie du restaurant Imagine un restaurant : le client (le navigateur) passe commande. Le serveur (le Contrôleur) prend la commande et coordonne. La cuisine + le frigo (le Modèle / la base de données) préparent les ingrédients. Le dressage de l'assiette (la Vue / Twig) rend le plat joli avant de le servir.

3Routing & Contrôleurs

Une route = une URL associée à une méthode PHP. Dans ton projet, on les déclare avec des attributs #[Route(...)] juste au-dessus des méthodes.

src/Controller/Front/HomeController.php 
class HomeController extends AbstractController
{
    #[Route('/', name: 'app_home')]
    public function index(PropertyRepository $propertyRepository): Response
    {
        // 1. On demande les annonces publiées au Repository
        // 2. On renvoie un template + les données
        return $this->render('front/home/index.html.twig', [
            'properties' => $propertyRepository->findForListing('published'),
        ]);
    }

    #[Route('/logement/{id}', name: 'app_logement_detail')]
    public function detail(Property $property): Response
    {
        // {id} dans l'URL → Symfony charge tout seul la Property correspondante !
        ...
    }
}

Décortiquons

ÉlémentSignification
'/logement/{id}'Le chemin de l'URL. {id} est un paramètre dynamique.
name: 'app_logement_detail'Le nom de la route. Sert à générer l'URL ailleurs sans réécrire le chemin.
Property $propertyParamConverter : Symfony voit {id}, va chercher la Property en base et te la donne directement. Magique 🪄

Préfixe sur la classe

Dans Admin/PropertyController, le #[Route('/property')] est posé sur la classe : c'est un préfixe commun. Toutes les routes en dessous commencent donc par /property.

final class PropertyController extends AbstractController

    // URL finale = /property/new
    ...
}

4Doctrine : parler à la base de données

Une entité = une table

Une entité est une classe PHP qui représente une table. Chaque propriété = une colonne. On le déclare avec des attributs #[ORM\...].

src/Entity/Property.php 
#[ORM\Entity(repositoryClass: PropertyRepository::class)]
#[ORM\Table(name: 'properties')]     // nom de la table
class Property
{
    #[ORM\Column(length: 255)]       // colonne VARCHAR(255)
    private ?string $title = null;

    #[ORM\Column(type: Types::DECIMAL, precision: 10, scale: 2)]
    private ?string $pricePerNight = null;  // prix → string !
}
⚠️ Pourquoi le prix est un string et pas un float ? Les float font des arrondis dangereux pour l'argent (0.1 + 0.2 ≠ 0.3 en informatique !). Le type DECIMAL garde la précision exacte, et PHP le manipule comme une chaîne pour ne pas perdre cette précision.

Enregistrer / supprimer : l'EntityManager

Pour modifier la base, on utilise l'EntityManagerInterface. Deux étapes clés : persist() puis flush().

src/Controller/Admin/PropertyController.php — action new() 
$property = new Property();
// ... on remplit l'objet via le formulaire ...

$entityManager->persist($property);  // 1. « prépare cet objet pour la sauvegarde »
$entityManager->flush();             // 2. « exécute vraiment le SQL (INSERT) »
🛒 Analogie du caddie persist() = mettre un article dans le caddie. flush() = passer en caisse (tout est enregistré d'un coup). Tu peux persist plusieurs objets puis un seul flush().
OpérationCode
Créerpersist($obj) + flush()
Modifierchanger l'objet + flush() (déjà connu de Doctrine)
Supprimerremove($obj) + flush()
Lire par id$repo->find($id)

Les tables sont liées entre elles. Doctrine connaît 4 types de relations. Dans ton projet, tout tourne autour de la Property.

👤 User(l'hôte)
1 — N
🏠 Property
N — 1

Les 4 types

TypeExemple dans ton projet
ManyToOnePlusieurs Property appartiennent à un User (hôte).
OneToManyUn User possède plusieurs Property. (côté inverse du précédent)
OneToOneUne Property a une seule PropertyAddress.
ManyToManyDes logements partagent plusieurs équipements (via PropertyAmenity).

Une relation a deux côtés

src/Entity/Property.php — côté « propriétaire » 
#[ORM\ManyToOne(inversedBy: 'properties')]
#[ORM\JoinColumn(nullable: false)]   // la colonne host_id est obligatoire
private ?User $host = null;
src/Entity/User.php — côté « inverse » 
#[ORM\OneToMany(targetEntity: Property::class, mappedBy: 'host')]
private Collection $properties;
Mot-cléRôle
inversedBy / mappedByRelient les deux côtés d'une relation bidirectionnelle.
cascade: ['persist','remove']Si je sauve/supprime la Property, son adresse suit automatiquement.
orphanRemoval: trueUn enfant retiré de la collection est supprimé de la base.
💡 Pourquoi new ArrayCollection() dans le constructeur ? Les côtés OneToMany/ManyToMany sont des collections (listes d'objets). On les initialise vides dans le __construct() pour pouvoir faire ->add() sans erreur.

6Les Repositories : les requêtes

Un Repository est la classe spécialisée pour lire les données d'une entité. C'est là qu'on écrit nos requêtes personnalisées avec le QueryBuilder.

src/Repository/PropertyRepository.php 
public function findForListing(?string $status = null): array
{
    $qb = $this->createQueryBuilder('p')
        ->addSelect('m', 'a', 'host')
        ->leftJoin('p.media', 'm')     // charge les photos
        ->leftJoin('p.address', 'a')   // charge l'adresse
        ->leftJoin('p.host', 'host')  // charge l'hôte
        ->orderBy('p.createdAt', 'DESC');

    if ($status !== null) {
        $qb->andWhere('p.status = :status')
            ->setParameter('status', $status);  // valeur sécurisée
    }

    return $qb->getQuery()->getResult();
}
⚠️ Le problème N+1 (très important !) Si tu affiches 20 logements et que pour chacun tu refais une requête pour l'adresse, la photo, l'hôte… ça fait 1 + 20×3 = 61 requêtes ! Lent.
👉 La solution : leftJoin() + addSelect() chargent tout en une seule requête. C'est exactement ce que fait findForListing().
❌ Sans JOIN61 requêtes
vs
✅ Avec JOIN1 requête
✅ Sécurité : setParameter() On ne colle jamais une valeur directement dans la requête. setParameter() la transmet séparément → protection contre les injections SQL.

7Formulaires & Validation

Symfony génère et traite les formulaires pour toi. Un FormType décrit les champs ; le contrôleur le branche sur une entité.

1) Décrire le formulaire

src/Form/PropertyType.php 
class PropertyType extends AbstractType
{
    public function buildForm(FormBuilderInterface $builder, array $options): void
    {
        $builder
            ->add('title', TextType::class, ['label' => 'Titre'])
            ->add('propertyType', ChoiceType::class, [
                'choices' => ['Villa' => 'villa', 'Loft' => 'loft'],
            ])
            ->add('pricePerNight', NumberType::class, ['scale' => 2]);
    }

    public function configureOptions(OptionsResolver $resolver): void
    {
        $resolver->setDefaults(['data_class' => Property::class]);
        // ↑ ce formulaire « remplit » un objet Property
    }
}

2) Le traiter dans le contrôleur

$property = new Property();
$form = $this->createForm(PropertyType::class, $property);
$form->handleRequest($request);  // lit les données envoyées et remplit $property

if ($form->isSubmitted() && $form->isValid()) {  // ← validation ici
    $entityManager->persist($property);
    $entityManager->flush();
    return $this->redirectToRoute('app_property_index');
}

3) La validation : les contraintes

Les règles de validité sont posées directement sur l'entité avec #[Assert\...]. Elles sont vérifiées au moment du isValid().

src/Entity/Property.php 
#[Assert\NotBlank(message: 'Le titre est obligatoire.')]
#[Assert\Length(min: 5, max: 255, minMessage: 'Au moins {{ limit }} caractères.')]
private ?string $title = null;

#[Assert\Positive(message: 'Le prix doit être supérieur à 0.')]
private ?string $pricePerNight = null;
ContrainteVérifie que…
#[Assert\NotBlank]le champ n'est pas vide
#[Assert\Length(min,max)]la longueur est correcte
#[Assert\Positive]le nombre est > 0
#[Assert\Choice([...])]la valeur fait partie d'une liste autorisée
#[Assert\GreaterThan(propertyPath:'checkinDate')]la date de départ est après l'arrivée
🔒 Protection CSRF Symfony ajoute automatiquement un jeton CSRF caché dans chaque formulaire. Il empêche un site malveillant de soumettre un formulaire à ta place. On le vérifie aussi manuellement pour les suppressions : isCsrfTokenValid('delete'.$id, $token).

Twig est le moteur de gabarits. Il mélange du HTML et des données, avec une syntaxe simple et 3 délimiteurs.

SyntaxeRôleExemple

💡 {{ property.title }} = appel intelligent Twig essaie dans l'ordre : la propriété publique title, puis getTitle(), puis isTitle(). Donc pas besoin d'écrire getTitle() à la main.

L'héritage de templates

On ne réécrit pas le <head>, le menu, le footer à chaque page. Un template parent définit la structure avec des {% block %} ; les enfants remplissent ces blocs.

templates/layout/base.html.twig (le parent) 
<!DOCTYPE html>
<html lang="fr">
  <head> ... </head>
  {% block body %}{% endblock %}   <!-- emplacement à remplir -->
</html>
templates/front/home/index.html.twig (l'enfant) 
{% extends 'layout/base.html.twig' %}   {# je réutilise le parent #}

{% block body %}
  <h1>Nos logements</h1>
  {% for property in properties %}
    <div>{{ property.title }} — {{ property.pricePerNight }} €</div>
  {% endfor %}
{% endblock %}

9La sécurité

Deux notions à ne pas confondre : l'authentification (« qui es-tu ? ») et l'autorisation (« as-tu le droit ? »).

🔑 Authentification« es-tu bien Marie ? »
login + mot de passe
puis
🛂 Autorisation« Marie peut-elle voir
cette page ? »

La configuration : security.yaml

config/packages/security.yaml 
providers:                 # OÙ trouver les utilisateurs
  users_in_database:
    entity:
      class: App\Entity\User
      property: email        # on se connecte avec l'email

firewalls:                # COMMENT on se connecte
  main:
    form_login: { login_path: app_login }
    logout: { path: app_logout }
    login_throttling: { max_attempts: 5 }  # anti-bruteforce

access_control:            # QUI accède à QUOI
  - { path: ^/admin, roles: ROLE_ADMIN }
  - { path: ^/reservations, roles: ROLE_USER }
  - { path: ^/, roles: PUBLIC_ACCESS }
💡 access_control : la première règle qui correspond gagne Symfony lit les règles de haut en bas et s'arrête à la première qui matche l'URL. Donc on met les règles précises (^/admin) avant les générales (^/).

La hiérarchie des rôles

security.yaml 
role_hierarchy:
  ROLE_HOST: ROLE_USER
  ROLE_ADMIN: [ROLE_MODERATEUR, ROLE_HOST]
  ROLE_SUPER_ADMIN: ROLE_ADMIN

Un ROLE_ADMIN hérite donc de ROLE_HOST, qui hérite de ROLE_USER. Pas besoin de tout lui donner manuellement.

ROLE_ADMIN
↓ contient

Le contrôle fin : les Voters

Un rôle, c'est trop large. Pour des règles sur un objet précis (« CET hôte possède-t-il CE logement ? »), on utilise un Voter.

src/Security/Voter/ReservationVoter.php 
protected function voteOnAttribute($attribute, $subject, $token): bool
{
    $user = $token->getUser();
    $reservation = $subject;

    if (in_array('ROLE_ADMIN', $user->getRoles())) return true;  // admin = OK

    $isGuest = $reservation->getGuest()?->getId() === $user->getId();
    $isHost = $reservation->getProperty()?->getHost()?->getId() === $user->getId();

    return match ($attribute) {
        self::VIEW => $isGuest || $isHost,   // voir : le voyageur OU l'hôte
        self::MANAGE => $isHost,                // gérer : seulement l'hôte
    };
}

On l'utilise ensuite comme un rôle : #[IsGranted(ReservationVoter::VIEW, subject: 'reservation')].

OutilQuand l'utiliser
Rôle (ROLE_ADMIN)Droit large : « est-ce un admin ? »
VoterDroit fin sur un objet : « est-ce SON logement à elle ? »

10Services & Injection de dépendances

Un service est une classe qui rend un service (logique métier, envoi de mail…). Tu ne l'instancies jamais toi-même : Symfony te le fournit automatiquement. C'est l'autowiring.

🔌 Analogie de la prise électrique Tu ne fabriques pas ta propre électricité : tu branches un appareil et le courant arrive. Avec l'autowiring, tu « déclares » qu'une méthode a besoin d'un PropertyRepository, et Symfony le « branche » tout seul.

En pratique : tu n'as qu'à le typer

public function index(PropertyRepository $propertyRepository): Response
{
    // Tu ne fais PAS : new PropertyRepository()
    // Symfony voit le type et te donne l'instance prête à l'emploi.
    $annonces = $propertyRepository->findForListing('published');
}
🎮 Contrôleur« j'ai besoin de PropertyRepository »
→
📦 Container Symfony« le voici, déjà construit »

Les Événements (EventSubscriber)

Symfony émet des événements à chaque étape (requête reçue, réponse envoyée…). Un EventSubscriber permet de réagir à un moment précis sans toucher au contrôleur.

public static function getSubscribedEvents(): array
{
    return [
        RequestEvent::class => 'onKernelRequest',  // quel event → quelle méthode
    ];
}
💡 Exemple : bloquer un utilisateur banni Le UserChecker bloque au moment de la connexion. Un Subscriber peut vérifier à chaque requête (même un utilisateur déjà connecté qui se fait bannir en cours de route).

11API Platform : la même entité en JSON

Avec un seul attribut, ton entité Property devient aussi une API REST (endpoints JSON automatiques), en plus du site web Twig.

src/Entity/Property.php 
#[ApiResource(
    operations: [
        new GetCollection(security: "is_granted('ROLE_USER')"),
        new Get(security: "is_granted('PROPERTY_VIEW', object)"),
        new Post(security: "is_granted('ROLE_ADMIN')"),
    ],
    normalizationContext: ['groups' => ['mon-groupe']],
)]
class Property { ... }

Cela crée automatiquement : GET /api/properties, GET /api/properties/{id}, POST /api/properties… chacun protégé par sa règle de sécurité.

Choisir les champs exposés : les Groups

On ne veut pas tout montrer en JSON. Les #[Groups] sélectionnent précisément les champs visibles (normalisation = objet → JSON).

#[Groups(['mon-groupe'])]          // CE champ sera dans le JSON
private ?string $title = null;
