Conception du module de recherche, reservation et calendrier iCal

1. Objectif du module

Le but de cette partie du projet est de realiser le coeur fonctionnel d'une plateforme de location type Airbnb pour le parcours principal suivant :

- rechercher un logement publie
- verifier sa disponibilite sur une periode
- calculer un prix coherent
- creer une reservation
- permettre a l'hote de confirmer ou refuser une demande
- permettre au voyageur ou a l'hote d'annuler une reservation
- gerer le calendrier du logement
- exporter et importer un calendrier iCal
- envoyer les notifications transactionnelles de facon asynchrone
- expirer automatiquement une demande pending apres 24h

La logique metier ne doit pas etre dispersee dans les controleurs. Les controleurs doivent surtout gerer la requete HTTP, les formulaires et la redirection. Les decisions metier importantes doivent etre centralisees dans des services dedies.

2. Principes d'architecture retenus

Nous retenons une architecture Symfony classique et lisible :

- les entites representent les donnees persistantes
- les repositories servent a lire les donnees en base
- les services portent les regles metier
- les formulaires encapsulent les saisies utilisateur
- les controleurs restent fins

Cette separation est importante pour eviter de dupliquer les verifications. La meme regle de disponibilite doit servir a la fois pour la recherche, pour le formulaire de reservation et pour la confirmation par l'hote.

3. Entites metier utilisees

Property

L'entite Property represente le logement. Elle contient les informations stables necessaires a la reservation :

- le statut du logement
- le nombre maximum de voyageurs
- le prix par nuit
- les frais de menage
- la caution
- les horaires d'arrivee et de depart
- le mode instant booking
- le token d'export iCal

Le logement est la racine du module : toutes les disponibilites, reservations et synchronisations iCal sont rattachees a lui.

Reservation

L'entite Reservation represente un sejour reserve ou demande. Elle contient :

- le logement
- le voyageur
- la date d'arrivee
- la date de depart
- le nombre de voyageurs
- le statut
- le prix total
- les frais calcules
- la devise
- le motif d'annulation si besoin

Les statuts actuellement utilises dans le projet sont :

- pending
- confirmed
- completed
- cancelled

Le statut cancelled couvre plusieurs cas metier :

- refus de l'hote
- annulation par le voyageur
- annulation par l'hote
- expiration automatique d'une demande pending

ReservationStatusHistory

Cette entite sert a garder une trace des changements de statut. Cela permet de savoir :

- quel etait l'ancien statut
- quel est le nouveau statut
- qui a fait l'action
- a quel moment le changement a ete effectue

Cette tracabilite est importante pour justifier les actions de l'hote et garder un historique clair.

PropertyAvailability

Cette entite represente les exceptions de calendrier jour par jour. Elle ne stocke pas tout le calendrier complet, seulement les jours qui ont une regle particuliere :

- un jour bloque
- un jour explicitement remis disponible
- un tarif personnalise
- une duree minimale de sejour
- une provenance du blocage

Le champ source permet de distinguer :

- manual : modification faite par l'hote
- ical_import : blocage importe depuis un calendrier externe

PropertyICalSync

Cette entite stocke les connexions vers des calendriers externes. Elle contient :

- le logement concerne
- le nom du fournisseur
- l'URL iCal distante
- la date de derniere synchronisation

Elle sert a relier proprement un import externe aux jours bloques qui en resultent.

4. Modelisation temporelle

Le systeme raisonne avec des intervalles de type [checkin, checkout).

Cela signifie :

- le jour d'arrivee est inclus
- le jour de depart est exclu

Concretement, une reservation du 10 au 12 occupe les nuits du 10 et du 11. Le 12 peut etre reserve par un autre voyageur. Cette regle correspond au fonctionnement habituel d'une plateforme de location courte duree.

Ce choix est essentiel pour eviter les faux conflits de reservation.

5. Regle de disponibilite

La disponibilite d'un logement repose sur une regle unique centralisee dans PropertyAvailabilityService.

Pour qu'un logement soit reservable, il faut que :

- le logement soit publie
- la date de depart soit strictement apres la date d'arrivee
- le nombre de voyageurs soit valide
- le nombre de voyageurs ne depasse pas la capacite maximale
- aucune reservation confirmee ne chevauche la periode
- aucun jour de la periode ne soit bloque
- la duree minimale de sejour soit respectee

Cette centralisation est un choix fort de conception. Sans cela, la recherche pourrait afficher un logement disponible alors que le checkout le refuserait ensuite.

6. Choix sur les statuts de reservation

Nous avons retenu le comportement suivant :

- si instant booking est actif, la reservation est creee directement en confirmed
- sinon, la reservation est creee en pending

Une reservation pending ne bloque pas automatiquement les dates. Ce blocage n'existe qu'a partir d'une reservation confirmed, ou d'un blocage manuel/externe dans le calendrier.

Ce choix est volontaire. Il evite qu'un simple voyageur monopolise un logement sans validation reelle. En contrepartie, plusieurs demandes pending peuvent exister sur une meme periode. Le controle final se fait donc au moment de l'acceptation par l'hote, avec une reverification complete de la disponibilite.

7. Creation d'une reservation

La creation d'une reservation est confiee a ReservationManager.

Ce service applique l'ordre logique suivant :

1. verifier que le voyageur ne reserve pas son propre logement
2. verifier la disponibilite complete
3. calculer le prix
4. creer la reservation avec le bon statut initial
5. enregistrer une premiere ligne d'historique
6. persister le tout en base

Ce choix permet d'avoir un seul point d'entree metier pour la creation d'une reservation.

8. Calcul du prix

Le calcul du prix est gere par BookingPricingService.

Le principe retenu est :

- prix de base du logement par nuit
- surcharge eventuelle par jour via priceOverride
- ajout des frais de menage
- calcul des frais de service
- conservation eventuelle de la caution

Le total affiche au voyageur et le total enregistre sur la reservation proviennent de la meme logique, ce qui evite les incoherences entre l'ecran de confirmation et la base.

9. Moderation par l'hote

L'hote doit pouvoir traiter les demandes pending. Cette logique est centralisee dans ReservationStatusManager.

Deux actions sont prevues :

- confirmer une demande pending
- refuser une demande pending

Lors d'une confirmation :

- la reservation doit encore etre pending
- la disponibilite est reverifiee au dernier moment
- le statut passe a confirmed
- un historique est ajoute

Lors d'un refus :

- la reservation doit encore etre pending
- un motif est obligatoire
- le statut passe a cancelled
- le motif est conserve
- un historique est ajoute

Cette conception protege le systeme contre le cas suivant : une demande etait valide au moment de sa creation, mais les dates ont ete prises entre temps.

10. Annulation d'une reservation

Le sujet demande aussi qu'une reservation puisse etre annulee par l'hote ou par le voyageur, avec motif obligatoire.

Nous avons donc retenu la regle suivante :

- une reservation pending peut etre annulee
- une reservation confirmed peut etre annulee
- une reservation completed ne peut plus etre annulee par ce flux
- un motif d'annulation est toujours stocke

La logique est centralisee dans ReservationStatusManager, afin d'eviter des comportements differents selon la page utilisee.

Lors d'une annulation :

- le statut passe a cancelled
- le motif est enregistre
- une ligne d'historique est ajoutee
- les dates sont liberees immediatement puisque seules les reservations confirmed bloquent reellement la disponibilite
- une notification asynchrone est envoyee aux parties concernees

Ce point est important : il n'est pas necessaire de supprimer la reservation pour liberer le calendrier. Il suffit de sortir la reservation de l'etat confirmed.

11. Gestion du calendrier par l'hote

La gestion manuelle du calendrier est prise en charge par PropertyAvailabilityManager.

L'hote peut appliquer une plage de dates avec :

- un mode blocked
- un mode available
- une duree minimale de sejour
- un tarif personnalise

Le choix de modeliser ces regles jour par jour dans PropertyAvailability est pertinent car :

- un tarif peut changer d'un jour a l'autre
- une duree minimale peut etre ponctuelle
- un blocage peut concerner seulement quelques dates

Quand l'hote remet une plage en disponible sans autre regle, les lignes manuelles inutiles sont supprimees. Cela evite de surcharger la base avec des entrees sans valeur metier.

12. Construction du calendrier visuel

L'affichage mensuel du calendrier repose sur PropertyAvailabilityCalendarBuilder.

Le calendrier croise deux sources :

- les reservations confirmed
- les overrides de PropertyAvailability

Chaque jour du calendrier est classe dans un etat principal :

- available
- booked
- blocked
- imported

Le calendrier peut aussi afficher :

- un tarif personnalise
- une contrainte de minimumStay

Ce choix de conception est utile pour l'interface hote, car il permet de distinguer visuellement :

- un jour reserve par un voyageur
- un jour bloque manuellement
- un jour bloque par import iCal

13. Conception de la recherche

La recherche frontale est centralisee dans PropertySearchService.

La logique retenue est la suivante :

- charger uniquement les logements publies
- filtrer par destination si elle est renseignee
- filtrer par capacite minimale
- si des dates sont donnees, reutiliser exactement la regle de disponibilite metier

Le point important est que la recherche ne repose pas sur une logique simplifiee differente. Elle appelle la meme verification que le module de reservation. Cela garantit une coherence entre :

- la liste des resultats
- la page logement
- le formulaire de reservation
- la confirmation par l'hote

14. Conception de l'export iCal

L'export iCal a ete pense comme un flux securise par token.

Chaque logement possede un iCalExportToken unique et regenerable. Le choix d'un token dedie est preferable a une URL basee uniquement sur l'identifiant du logement, car :

- l'URL est difficile a deviner
- l'hote peut invalider un ancien lien
- plusieurs plateformes externes peuvent consommer le flux sans compte local

Le flux exporte uniquement les reservations confirmed a venir. Cela correspond au besoin reel : informer les calendriers externes des nuits deja prises.

Les evenements iCal contiennent les informations minimales utiles :

- UID
- DTSTART
- DTEND
- SUMMARY
- DESCRIPTION
- STATUS

15. Conception de l'import iCal

L'import iCal permet de bloquer localement des dates venant d'une autre plateforme.

Le fonctionnement retenu est le suivant :

1. l'hote enregistre un fournisseur et une URL iCal
2. une synchronisation lit le fichier distant
3. les evenements sont traduits en jours bloques
4. ces jours sont stockes dans PropertyAvailability avec source = ical_import
5. les anciennes lignes importees de cette meme source sont mises a jour ou supprimees selon le nouveau contenu

Ce choix presente plusieurs avantages :

- l'import reste lisible en base
- chaque source iCal garde sa propre trace
- la suppression d'une source n'efface pas les reglages manuels
- les blocages externes sont reutilises par la recherche et la reservation sans logique speciale supplementaire

La synchronisation est actuellement declenchee :

- soit depuis l'interface hote
- soit via la commande app:property:ical:sync

Donc la mise a jour n'est pas automatique en continu. Si plus tard on veut une synchronisation reguliere, il suffira d'executer cette commande via cron ou scheduler.

16. Concurrence et prevention du double booking

Le cas le plus sensible est celui de deux utilisateurs qui tentent de finaliser en meme temps la meme periode pour le meme logement.

Sans protection supplementaire, le risque est le suivant :

1. les deux requetes lisent un etat encore disponible
2. les deux pensent pouvoir confirmer
3. les deux ecrivent presque en meme temps
4. on obtient un double booking

La bonne reponse de conception est donc la suivante :

- la validation finale ne doit pas reposer uniquement sur une verification faite avant paiement
- il faut refaire une verification juste au moment critique
- cette verification finale doit idealement etre executee dans une transaction base de donnees

Le scenario cible est :

1. ouvrir une transaction
2. relire le logement et les reservations concernees avec verrouillage d'ecriture
3. reexecuter la verification de chevauchement
4. si un conflit apparait, annuler la transaction
5. sinon, enregistrer la reservation ou la confirmation puis valider la transaction

Dans l'etat actuel du projet, nous avons deja une reverification metier au moment de la confirmation d'une demande pending par l'hote. Cela reduit le risque fonctionnel. En revanche, pour gerer de facon totalement robuste deux validations simultanees a la milliseconde pres, l'evolution naturelle est d'ajouter une transaction Doctrine avec verrouillage pessimiste lors de la phase finale de confirmation/paiement.

17. Performance de la verification de disponibilite

Verifier 20 nuits consecutives ne doit pas produire 20 requetes SQL.

Le choix retenu est de travailler par plage de dates avec un nombre fixe de requetes, independant du nombre de nuits :

- une requete pour savoir s'il existe deja une reservation confirmed qui chevauche la periode
- une requete pour charger les overrides de PropertyAvailability sur toute la plage
- puis un traitement en memoire pour analyser les jours bloques, les tarifs personnalises et la duree minimale de sejour

Autrement dit, on ne demande pas a la base :

- "le 10 est-il libre ?"
- puis "le 11 est-il libre ?"
- puis "le 12 est-il libre ?"

On lui demande plutot :

- "donne-moi tous les conflits de reservation sur cette plage"
- "donne-moi toutes les exceptions de calendrier sur cette plage"

Cette approche est bien plus efficace car :

- le nombre de requetes reste stable
- la base filtre les donnees utiles
- le PHP ne traite que les lignes pertinentes

Le calcul du prix suit la meme logique : on charge les overrides de la plage une seule fois, puis on calcule les nuits en memoire.

Pour tenir la charge sur un vrai volume de donnees, il est egalement pertinent d'avoir des index adaptes sur les champs de recherche temporelle, notamment :

- reservations : logement + dates
- property_availability : logement + available_date

18. Asynchronisme et notifications

L'envoi d'email doit preferentiellement intervenir apres le flush() de l'ORM.

La raison est simple : avant le flush, l'etat metier n'est pas encore garanti en base. Si on envoie un email trop tot, on peut envoyer :

- une confirmation pour une reservation qui sera finalement rollback
- une notification avec un identifiant non encore persiste
- un message incoherent par rapport a l'etat reel de la base

Le bon enchainement conceptuel est donc :

1. creer ou modifier la reservation
2. flush les donnees
3. seulement apres cela, declencher la notification asynchrone

Concernant la panne de RabbitMQ, la strategie doit toujours prevoir un filet de securite. Dans ce projet, la configuration Messenger actuelle utilise deja un transport Doctrine, avec retries et failure transport. C'est une bonne base de resilience car :

- les messages peuvent rester stockes en base
- les echecs temporaires peuvent etre retentes
- les messages en echec peuvent etre isoles puis retraites

Donc la reponse attendue est la suivante :

- si RabbitMQ etait utilise et tombait en panne, il ne faudrait pas perdre les notifications
- il faut prevoir un transport de secours ou un stockage persistant
- les handlers doivent rester rejouables
- les messages definitivement en echec doivent aller dans une file failed

Dans l'etat actuel du projet, le choix du transport Doctrine repond deja bien a cette exigence de robustesse pour l'environnement de developpement et pour une premiere version du sujet.

Concretement, nous avons separe deux responsabilites :

- ReservationNotificationDispatcher decide quel message asynchrone envoyer apres une creation, une confirmation, un refus, une annulation ou une expiration
- les handlers Messenger envoient ensuite les emails vers Mailpit

Cette separation est saine car la logique metier n'envoie pas directement les emails. Elle se contente de decrire l'evenement metier survenu.

19. Expiration automatique des demandes pending

Le bonus G.1 demande qu'une reservation pending expire automatiquement apres 24h si elle n'est pas confirmee.

Le mecanisme retenu est le suivant :

1. a la creation d'une reservation pending, on envoie immediatement un message de notification a l'hote
2. dans le meme temps, on planifie un message ExpirePendingReservationMessage avec un delai de 24h
3. lorsque ce message est consomme par le worker, on reverifie que la reservation est toujours pending
4. on reverifie aussi que son createdAt est bien plus ancien que 24h
5. si ces conditions sont toujours vraies, la reservation passe a cancelled avec un motif systeme
6. une ligne d'historique est ajoutee avec changedBy a null
7. une notification est envoyee aux deux parties

Ce schema est robuste car :

- si l'hote confirme avant les 24h, le handler ne fera rien
- si le voyageur annule avant les 24h, le handler ne fera rien
- le worker ne depend pas d'un cron externe pour savoir quoi faire reservation par reservation

20. Priorite des sources de blocage

L'ordre metier retenu est le suivant :

- une reservation confirmed prime comme indisponibilite reelle
- un blocage manuel bloque la date
- un blocage iCal importe bloque aussi la date
- une reservation pending ne bloque pas la date

Cette hierarchie est coherente avec le besoin du sujet : ne pas autoriser de double reservation, tout en conservant de la souplesse pour les demandes non encore validees.

21. Coherence globale du systeme

Le coeur de la conception repose sur une idee simple : une seule verite metier pour la disponibilite.

Cette verite est ensuite reutilisee partout :

- dans la recherche
- dans la creation d'une reservation
- dans la moderation par l'hote
- dans le calendrier visuel
- dans les imports iCal

Ce choix rend le module plus fiable, plus facile a maintenir et plus proche d'un comportement attendu sur une vraie plateforme de location.

22. Conclusion

Pour cette partie du projet, nous avons retenu une conception volontairement propre et evolutive :

- les reservations restent stockees sous forme de periodes
- les exceptions de calendrier sont stockees jour par jour
- la verification de disponibilite est centralisee
- la moderation hote recontrole la disponibilite avant confirmation
- l'annulation voyageur ou hote conserve un motif et libere immediatement les dates
- les emails transactionnels passent par Messenger puis Mailpit
- les demandes pending peuvent expirer automatiquement apres 24h
- l'iCal exporte les sejours confirmes
- l'iCal importe des blocages externes reutilisables par tout le systeme

Cette conception repond bien au besoin principal du sujet pour le bloc recherche, reservation et synchronisation de calendrier, tout en restant compatible avec les futures evolutions du projet.
