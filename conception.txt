Conception technique - Module reservations et calendrier
========================================================

Projet : clone Airbnb Symfony
Objectif : concevoir et implementer un moteur de reservation capable de gerer les disponibilites, les demandes en attente, la confirmation hote, les blocages manuels et les conflits de concurrence.


1. Périmètre fonctionnel traité
-------------------------------

Le module couvre actuellement les points suivants :

- creation d'une demande de reservation depuis une fiche logement ;
- calcul du prix cote serveur ;
- verification des disponibilités avant creation ;
- historique voyageur des reservations ;
- detail sécurisé d'une reservation ;
- gestion des demandes pending par l'hôte ;
- acceptation ou refus d'une demande par l'hôte ;
- blocage manuel de dates par l'hôte ;
- prevention des doubles reservations confirmees en base de donnees.

Les sujets suivants restent a completer dans une phase suivante :

- notifications email asynchrones via Messenger et Mailer ;
- paiement Stripe ;
- import/export iCal ;
- recherche filtree strictement par disponibilite ;
- tests automatises complets.


2. Regle temporelle retenue
---------------------------

Une reservation est representee par deux dates :

- checkinDate : jour d'arrivee inclus ;
- checkoutDate : jour de depart exclu.

La periode est donc traitee comme un intervalle [checkin, checkout).

Exemple :

Une reservation du 10 au 13 bloque les nuits du 10, 11 et 12. Le 13 est de nouveau reservable par un autre voyageur.

Ce choix correspond au fonctionnement hotelier classique : un voyageur quitte le logement le matin du checkout, et un autre peut arriver le meme jour plus tard. Il evite aussi de bloquer inutilement une nuit qui n'est pas consommee.

Cette règle est utilisée partout :

- calcul du nombre de nuits ;
- verification des chevauchements ;
- contrainte PostgreSQL ;
- blocage manuel des dates par l'hôte.


3. Cycle de vie d'une reservation
---------------------------------

Les statuts utilisés sont :

- pending : demande envoyée, en attente de validation hôte ;
- confirmed : reservation acceptee, dates verrouillées ;
- cancelled : reservation refusée ou annulée ;
- completed : séjour termine.

Le statut initial depend du paramétrage du logement :

- instantBooking = true : la reservation est créée directement en confirmed ;
- instantBooking = false : la reservation est créée en pending.

Ce choix permet de gérer les deux cas demandes dans le sujet : reservation instantanée et reservation sur demande.

Chaque creation ou changement de statut généré une entree ReservationStatusHistory. L'objectif est de garder une trace de la decision, de l'ancien statut, du nouveau statut et de l'utilisateur qui a effectue l'action.


4. Choix sur les reservations pending
-------------------------------------

Les reservations pending ne bloquent pas les dates.

Seules les reservations confirmed bloquent effectivement le calendrier.

Ce choix est volontaire. Une demande pending n'est pas encore une garantie pour le voyageur : elle doit etre acceptee par l'hote. Si les pending bloquaient les dates, un utilisateur pourrait bloquer abusivement un logement sans paiement ni validation.

Consequence :

Plusieurs demandes pending peuvent exister pour le meme logement et les memes dates.

Regle d'arbitrage :

La premiere demande qui passe en confirmed gagne. Les autres ne peuvent plus etre acceptees si elles chevauchent la reservation confirmee.

Au moment ou l'hote accepte une demande, le systeme reverifie la disponibilite. Si les dates ont ete confirmees entre-temps pour une autre reservation, l'acceptation echoue avec un message explicite.


5. Detection des chevauchements
-------------------------------

La formule retenue pour detecter un chevauchement est :

existing.checkinDate < requestedCheckout
AND existing.checkoutDate > requestedCheckin

Cette formule est fiable pour des intervalles [debut, fin). Elle permet bien le cas suivant :

- reservation A : 10 -> 13 ;
- reservation B : 13 -> 15 ;
- pas de chevauchement.

En revanche, ce cas est refuse :

- reservation A : 10 -> 13 ;
- reservation B : 12 -> 15 ;
- chevauchement sur la nuit du 12.


6. Gestion de la concurrence
----------------------------

Une simple verification PHP ne suffit pas pour eviter les doubles reservations. Deux utilisateurs peuvent soumettre une reservation en meme temps :

1. utilisateur A verifie la disponibilite : OK ;
2. utilisateur B verifie la disponibilite : OK ;
3. A insere une reservation confirmed ;
4. B insere aussi une reservation confirmed.

Pour eviter ce probleme, la garantie finale est placee dans PostgreSQL.

Une migration cree l'extension btree_gist et une contrainte d'exclusion :

EXCLUDE USING gist (
    property_id WITH =,
    daterange(checkin_date, checkout_date, '[)') WITH &&
)
WHERE (status = 'confirmed')

Cette contrainte interdit deux reservations confirmed qui se chevauchent pour un meme logement.

Pourquoi ce choix :

- la base de donnees est le dernier arbitre ;
- la solution reste correcte meme sous forte concurrence ;
- elle respecte directement la regle [checkin, checkout) ;
- elle ne bloque pas inutilement les demandes pending.

Le code applicatif utilise aussi une transaction via EntityManager::wrapInTransaction() lors de la creation ou de l'acceptation d'une reservation. Si PostgreSQL refuse l'insertion a cause de la contrainte, l'erreur est transformee en message metier :

"Ces dates viennent d'etre reservees. Choisissez une autre periode."


7. Services metier
------------------

La logique de reservation n'est pas placee directement dans les controleurs. Elle est decoupee en services.

AvailabilityChecker
-------------------

Responsabilite : verifier si une periode est reservable.

Regles controlees :

- le logement doit etre publie ;
- la date d'arrivee ne peut pas etre passee ;
- la date de depart doit etre posterieure a la date d'arrivee ;
- le nombre de voyageurs doit etre positif ;
- le nombre de voyageurs ne doit pas depasser la capacite du logement ;
- aucune nuit de la periode ne doit etre bloquee manuellement ;
- aucune reservation confirmed ne doit chevaucher la periode.

BookingPriceCalculator
----------------------

Responsabilite : calculer le prix d'une reservation.

Le calcul actuel est :

total = prix des nuits + frais de menage + frais de service + caution

Le prix des nuits est calcule nuit par nuit. Si une ligne PropertyAvailability possede un priceOverride pour une date, ce prix remplace le prix standard du logement pour cette nuit.

Les frais de service sont fixes a 12 % du montant des nuits.

Ce choix est un compromis MVP : il fournit un calcul clair et centralise, mais la valeur 12 % devra idealement devenir un parametre configurable.

La devise est actuellement EUR. Ce choix simplifie le MVP mais devra etre aligne plus tard avec preferredCurrency ou une configuration globale.

BookingManager
--------------

Responsabilite : creer une reservation.

Il orchestre :

- verification de disponibilite ;
- calcul du prix ;
- choix du statut initial ;
- creation de Reservation ;
- creation de ReservationStatusHistory ;
- transaction ;
- gestion propre des conflits PostgreSQL.

ReservationStatusManager
------------------------

Responsabilite : gerer les transitions faites par l'hote.

Il permet :

- d'accepter une reservation pending ;
- de refuser une reservation pending avec motif ;
- de creer l'historique de statut.

Lors de l'acceptation, la disponibilite est reverifiee. C'est indispensable car une demande pending ne bloque pas les dates.

PropertyAvailabilityManager
---------------------------

Responsabilite : gerer les indisponibilites manuelles du calendrier.

Il permet :

- de bloquer une periode ;
- de debloquer une date ;
- de creer une ligne PropertyAvailability par nuit.


8. Choix de modelisation du calendrier
--------------------------------------

Le projet possedait deja l'entite PropertyAvailability. Elle a ete reutilisee au lieu de recreer un nouveau modele.

Structure utilisee :

- property : logement concerne ;
- availableDate : date precise ;
- isAvailable : true ou false ;
- priceOverride : tarif specifique optionnel ;
- minimumStay : duree minimale optionnelle.

Le choix actuel est de stocker une ligne par jour.

Avantages :

- requetes simples ;
- blocage precis d'une nuit ;
- tarif specifique facile a appliquer ;
- affichage simple dans l'interface hote.

Limites :

- pour de tres longues periodes, cela cree beaucoup de lignes ;
- une table de periodes avec startDate/endDate serait plus compacte ;
- le minimumStay est stocke mais pas encore applique dans AvailabilityChecker.

Pour un MVP pedagogique, la table de jours individuels est acceptable car elle rend les regles faciles a comprendre et a tester.


9. Interface voyageur
---------------------

La fiche logement contient maintenant un formulaire simple :

- date d'arrivee ;
- date de depart ;
- nombre de voyageurs.

Ce formulaire ne cree pas directement la reservation. Il redirige vers une page de finalisation.

Pourquoi une page de finalisation :

- afficher un recapitulatif avant validation ;
- recalculer le prix cote serveur ;
- rappeler si la reservation sera immediate ou soumise a validation hote ;
- eviter de creer une reservation par simple clic accidentel.

Le voyageur ne peut pas choisir le statut, le prix ou l'utilisateur rattache a la reservation. Ces donnees sont decidees cote serveur.


10. Interface hote - demandes pending
-------------------------------------

Une interface hote a ete ajoutee sur :

/host/reservations

Elle affiche uniquement les demandes pending concernant les logements de l'hote connecte.

L'hote peut :

- accepter une demande ;
- refuser une demande avec motif.

L'acces est protege par ROLE_HOST.

Un voyageur simple ne peut pas acceder a cette interface. Une tentative d'acces retourne une erreur 403.


11. Interface hote - calendrier
-------------------------------

Une interface hote a ete ajoutee sur :

/host/calendar

Elle liste les logements de l'hote.

Pour chaque logement, l'hote peut ouvrir :

/host/properties/{id}/calendar

Cette page permet :

- de bloquer une periode ;
- de renseigner optionnellement un prix specifique par nuit ;
- de renseigner optionnellement une duree minimale ;
- de voir les nuits actuellement bloquees ;
- de rendre une nuit de nouveau disponible.

Le blocage utilise aussi la regle [debut, fin). Bloquer du 10 au 13 bloque les nuits du 10, 11 et 12.


12. Securite applicative
------------------------

Les routes voyageurs de reservation sont protegees par ROLE_USER.

Les routes hote sont protegees par ROLE_HOST.

Les routes administrateur restent protegees par ROLE_ADMIN.

Le detail d'une reservation voyageur utilise une requete specifique :

- la reservation doit appartenir au voyageur connecte ;
- sinon la page retourne une 404.

Ce choix evite qu'un utilisateur puisse voir une reservation d'un autre voyageur en devinant son UUID.

Les actions POST hote utilisent des tokens CSRF :

- accepter une demande ;
- refuser une demande ;
- debloquer une date.


13. Repositories
----------------

Les repositories restent limites aux requetes Doctrine.

ReservationRepository contient notamment :

- findForGuest() ;
- findOneForGuestDetail() ;
- findPendingForHost() ;
- findOnePendingForHost().

PropertyRepository contient :

- findForHost() ;
- findOneForHost().

PropertyAvailabilityRepository contient :

- findOneForPropertyDate() ;
- findBlockedForProperty() ;
- findOneForProperty().

Ce decoupage permet aux controleurs de rester plus lisibles et evite les acces directs non securises.


14. Choix sur les erreurs utilisateur
-------------------------------------

Les erreurs metier sont levees sous forme de DomainException dans les services.

Les controleurs les attrapent pour afficher un message comprehensible :

- dates deja reservees ;
- logement indisponible ;
- motif de refus manquant ;
- demande qui n'est plus en attente.

Ce choix separe les erreurs techniques des erreurs fonctionnelles.


15. Limites actuelles et ameliorations prevues
----------------------------------------------

Concurrence :

La contrainte PostgreSQL protege les reservations confirmed. C'est robuste. En revanche, les pending multiples sont acceptes volontairement. Il faudra clairement l'expliquer : une demande pending n'est pas une garantie.

Paiement :

Le paiement Stripe n'est pas encore branche. Quand il le sera, il faudra decider si la reservation passe en confirmed avant ou apres paiement.

Notifications :

Les emails ne sont pas encore envoyes. La prochaine etape est d'ajouter Messenger :

- notification hote lors d'une nouvelle demande pending ;
- notification voyageur lors de l'acceptation ;
- notification voyageur lors du refus ;
- notification des deux parties lors d'une annulation.

Recherche :

La recherche doit encore etre connectee au meme moteur de disponibilite. Elle devra exclure les logements deja confirmes sur les dates demandees et ceux dont la capacite est insuffisante.

iCal :

L'export et l'import iCal sont maintenant couverts au niveau MVP.

Export :

- chaque logement peut avoir un token prive stocke sur Property ;
- l'URL /api/properties/{id}/calendar.ics?token=... n'exige pas de session, car elle doit etre lisible par des clients calendrier externes ;
- le token est obligatoire et compare avec hash_equals ;
- l'hote peut generer, regenerer ou revoquer le token depuis son calendrier ;
- seules les reservations confirmed sont exportees ;
- les dates sont exportees en evenements jour entier, avec la meme convention [checkin, checkout).

Import :

- les flux externes sont stockes dans PropertyICalSync avec providerName, iCalUrl et lastSyncAt ;
- l'hote ajoute ou supprime ses flux depuis /host/properties/{id}/calendar ;
- la commande app:ical:sync lit tous les flux et transforme les evenements futurs en nuits indisponibles via PropertyAvailability ;
- les evenements passes sont ignores et les evenements deja commences sont tronques a aujourd'hui ;
- l'import est additif : il bloque ou remet a jour les nuits recues, mais ne supprime pas automatiquement les anciens blocages.

Le choix additif est volontaire pour le MVP. Le modele PropertyAvailability ne distingue pas encore une indisponibilite manuelle d'une indisponibilite importee depuis un flux donne. Supprimer automatiquement des dates absentes d'un flux pourrait donc effacer un blocage manuel de l'hote. Pour aller plus loin, il faudrait ajouter une origine au blocage, par exemple source = manual|ical, sourceSync et externalUid.

En cas de conflit, une reservation locale confirmed reste prioritaire. L'import iCal ne doit jamais annuler automatiquement une reservation confirmee.

Tests :

Le projet ne contient pas encore de tests automatises. Les tests prioritaires a ajouter sont :

- checkout exclusif ;
- chevauchement refuse ;
- chevauchement autorise au jour de depart ;
- pending ne bloque pas ;
- confirmed bloque ;
- acceptation hote impossible si les dates sont prises ;
- voyageur interdit de consulter la reservation d'un autre ;
- hote interdit de gerer le calendrier d'un autre hote.


16. Synthese des choix principaux
---------------------------------

Les decisions les plus importantes sont :

- intervalle [checkin, checkout) pour respecter le fonctionnement hotelier ;
- pending ne bloque pas pour eviter les blocages abusifs ;
- confirmed bloque de maniere stricte ;
- PostgreSQL garantit l'absence de double reservation confirmed ;
- logique metier dans des services reutilisables ;
- controleurs fins ;
- calendrier stocke par nuit via PropertyAvailability ;
- securisation par roles et requetes filtrees par utilisateur/hote ;
- historique systematique des changements de statut.

Ces choix privilegient un MVP robuste et comprehensible, tout en gardant une architecture extensible pour les prochaines parties du sujet : notifications, recherche avancee, paiement et iCal.
