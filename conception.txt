===============================================

1. DISPONIBILITÉS : PÉRIODES VS JOURS INDIVIDUELS
--------------------------------------------------

Pour représenter les jours indisponibles d'un logement, j'avais le choix
entre stocker une ligne par jour ou une ligne par période (date_debut / date_fin).

J'ai choisi les périodes. La raison principale : si un hôte bloque ses
vacances (ex. 3 semaines en août), ca fait 1 ligne au lieu de 21. Et la
condition SQL de chevauchement est simple :

    start_date < checkout  AND  end_date > checkin

Cette condition couvre tous les cas d'overlap possible en une seule expression,
sans boucle PHP nuit par nuit.

L'inconvénient est que l'affichage du calendrier mensuel nécessite de
parcourir les périodes en PHP pour colorier chaque case. Mais sur un logement
normal (quelques dizaines de réservations par an) ca reste très acceptable.


2. ÉTAT PENDING NE BLOQUE PAS LES DATES
-----------------------------------------

J'ai choisi que les réservations PENDING ne bloquent pas les dates.

Si le simple fait d'envoyer une demande bloquait le logement, n'importe quel
voyageur pourrait "geler" un logement 24h sans vraiment vouloir réserver.
Ca créerait des abus et bloquerait l'hôte inutilement.

Du coup deux voyageurs peuvent avoir des demandes en attente sur les mêmes
dates en même temps. Quand l'hôte confirme l'un, il faut refuser l'autre.
Le BookingService re-vérifie la disponibilité au moment de la confirmation
pour gérer ce cas. Et la commande app:expire-bookings expire automatiquement
les demandes non traitées après 24h.


3. RECHERCHE AVEC NOT EXISTS
------------------------------

Pour la page de recherche, je filtre les logements disponibles directement
en SQL avec des sous-requêtes NOT EXISTS en DQL. Ca évite de charger tous
les logements en mémoire et de filtrer en PHP.

La requête vérifie :
- Pas de réservation CONFIRMED qui chevauche les dates demandées
- Pas de période bloquée qui chevauche les dates demandées
- Capacité >= nombre de voyageurs

Un seul aller-retour base de données, peu importe la volumétrie.


4. NOTIFICATIONS APRÈS LE FLUSH
---------------------------------

Les emails se font APRÈS le flush() Doctrine, jamais avant.

Si j'envoie un email de confirmation avant que la réservation soit persistée
et que le flush() échoue ensuite, le voyageur reçoit un email pour une
réservation qui n'existe pas. C'est une incohérence critique.

J'utilise Symfony Messenger avec un transport "doctrine" : les messages sont
stockés dans la table messenger_messages. Même si le worker est arrêté, les
messages restent en base et sont traités au prochain démarrage.


5. ARCHITECTURE : SERVICES + DTOs + REPOSITORIES
--------------------------------------------------

Pour garder des contrôleurs fins et une logique réutilisable j'ai découpé
en plusieurs couches :

- AvailabilityService : vérifie la disponibilité sans effets de bord.
  Utilisé par la recherche ET lors de la création/confirmation d'une resa.

- BookingService : orchestre création, confirmation, annulation.
  Dispatche les messages Messenger après le flush().

- ICalService : génère le fichier .ics (RFC 5545) à partir des réservations
  confirmées. Appelé uniquement par ICalController.

- DTOs : BookingRequestDto encapsule les dates + voyageurs entre les étapes
  du tunnel de réservation (évite de dupliquer le parsing). SearchCriteriaDto
  regroupe les critères de recherche passés au template. AdminStatsDto
  transporte les agrégats du dashboard (counts SQL, pas de findAll()).

- Repositories : toutes les requêtes complexes sont dans le repository
  correspondant, pas dans le contrôleur ni dans le service.


6. TOKEN ICAL : SÉCURITÉ SANS SESSION
----------------------------------------

L'endpoint /api/properties/{id}/calendar.ics?token=... doit être accessible
par Google Calendar ou Outlook qui n'ont pas de session Symfony. J'utilise
donc un token dans l'URL à la place d'une authentification classique.

Le token est 64 caractères hexadécimaux (32 octets via random_bytes()),
unique par logement, stocké en base, révocable par l'hôte. La comparaison
utilise hash_equals() pour éviter les timing attacks.

Seules les réservations CONFIRMED sont exportées. Les demandes PENDING et
les données personnelles des voyageurs ne sont pas exposées dans le .ics.


ENTITÉS PRINCIPALES
--------------------
Property  → instantBooking (bool), calendarToken (varchar 64)
PropertyAvailability → startDate, endDate, reason (nullable)
Booking   → status (enum PENDING/CONFIRMED/CANCELLED/COMPLETED)
           → cancellationReason (text nullable)
User      → role (enum TRAVELER/HOST/ADMIN)
